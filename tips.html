Tips & Tricks

Getting the most out of the four text options

A closer look at tw, timeout, displaytime, and nodisplay — what each actually does at runtime, how they combine, and a couple of gotchas worth knowing before you lean on them.

The four options, recapped

Type thisSaved asWhereEffect at runtime
TW [tw] Any node's body text The line animates in letter-by-letter instead of appearing instantly.
TO 3 [timeout:3] Any node's body text The dialogue box is hidden for 3 seconds before the line shows at all, then appears and line_delivered fires.
nodisplay [nodisplay] Any node's body text The line is never shown and line_delivered never fires for this node — useful for a node that only exists to set up choices/conditions silently.
DT 2 [displaytime:2] Conditions node body text only The node waits 2 seconds, then auto-evaluates its branches and advances — this is the timer the countdown label in the dialogue box counts down.

See Editor Guide for exactly how typing these shortcuts works in the graph editor.

How they combine

All four can appear on the same line at once — they're independent tags, not alternatives. A Conditions node's body text can legally end up as something like "...[displaytime:4] [timeout:1] [tw]": the box stays hidden for 1 second, then the line types itself out, and 4 seconds after the node started processing, the conditions are evaluated and it moves on regardless of whether the typewriter animation has finished.

  • timeout delays the line appearing at all; tw only affects how it appears once it does. Combine them for a "beat of silence, then type it out" feel.
  • displaytime is independent of timeout and tw — it's timed from when the Conditions node starts processing, not from when the line finishes displaying or finishes typing. A short displaytime combined with a long timeout or a slow typewriter reveal can mean the node auto-advances before the player has even seen the full line — worth previewing in-editor before shipping a combination like that.
  • nodisplay skips the line and the timers that gate it showing (timeout, tw) but does not skip displaytime on a Conditions node — the node still silently waits out its displaytime before auto-advancing, it just never shows anything while doing so. This combination (nodisplay + displaytime) is a clean way to build an invisible "wait N seconds, then branch" node.
  • A Conditions node with no displaytime tag advances essentially immediately once ready — there's no default wait. Give it a DT value if you want the countdown label (and the pause before branching) to actually be visible/felt.

Authoring gotchas

Only one new shortcut converts per edit

If you type more than one raw shortcut (like TW and TO 3) in the same edit, only the last one you typed converts — the rest stay as plain text until you edit the line again. Already-converted bracket tags ([tw], [timeout:3], etc.) aren't affected by this and stack freely. See Editor Guide for the full explanation and an example.

Case and sign matter

TW, TO, and DT must be uppercase — lowercase versions are left as plain text. nodisplay is the one exception and must stay lowercase. None of the numeric options accept a minus sign; TO -1 or DT -1 just don't convert.

Keep tagged lines at least a few words long

The runtime looks for these tags by checking a small fixed window of trailing words in the node's text. A line that's very short — fewer than four space-separated words total, tags included — can fall outside that window and either fail to have its tag recognized, or in the shortest cases, error when the runtime tries to read a word position that doesn't exist. If you're using any of these tags, it's safest to pad the line out to at least a handful of words (or add a few words of filler if the line is naturally very short) rather than relying on a two- or three-word line plus a tag.

Put multiple tags at the very end, together

Whichever tags a line has always get kept together at the trailing edge of the text once you edit it (with [tw] normalized to come last). Avoid manually placing a tag in the middle of a sentence in the underlying JSON if you're hand-editing the file instead of using the graph editor — the runtime's own tag detection, like the editor's, expects them clustered at the end of the line.

Suggested patterns

  • Dramatic reveal: [timeout:1] [tw] — a short pause, then the line types itself out.
  • Silent setup node: [nodisplay] on a plain Dialogue node whose only job is to flow straight into the next node — nothing shown, no pause.
  • Timed decision window: on a Conditions node, [displaytime:5] [tw] — the line types out, the countdown label ticks down from 5 seconds, and whichever branch's condition is true when it hits zero gets taken.
  • Invisible timed branch: [nodisplay] [displaytime:3] on a Conditions node — nothing is shown to the player at all, but the node still waits 3 seconds before evaluating and advancing; handy for pacing a branch that shouldn't visibly interrupt whatever's on screen.
