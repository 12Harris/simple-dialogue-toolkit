Dialogue Format

Every dialogue is one plain JSON file: an object keyed by node id, where each node is exactly one of a line, a choice list, or a condition list.

Node shape

Every node has a display_name, a speaker, and text, plus exactly one of:

  • "next": "<id>" — a plain, unconditional link to the next node (or null to end the dialogue there).
  • "choices": [...] — player-facing options, each with a label, a next id, and an optional if condition (only shown as a clickable option if the condition passes).
  • "conditions": [...] — auto-routed branches, each with an if condition and a next id; the last entry (no if) is the fallback/else case.

Example — a plain line

"start": {
  "display_name": "Gate Challenge",
  "speaker": "Guard",
  "text": "Halt! State your business at the castle gate.",
  "next": "player_options"
}

Example — a choice node

"player_options": {
  "display_name": "Player Response Choice",
  "speaker": "Player",
  "text": "How will you handle this?",
  "choices": [
    { "label": "I am a knight of the realm", "next": "knight_check" },
    { "label": "Show my merchant permit.", "if": "has_permit", "next": "merchant_enter" },
    { "label": "Sweet-talk the guard.", "if": "charisma >= 8", "next": "charm_success" },
    { "label": "Turn back for now.", "next": "farewell" }
  ]
}

Example — a conditions node

"knight_check": {
  "display_name": "Knight Credential Check",
  "speaker": "Guard",
  "text": "A knight? I'll need to verify your credentials.",
  "conditions": [
    { "if": "has_seal", "next": "knight_enter" },
    { "if": "reputation >= 5", "next": "knight_trusted" },
    { "next": "knight_denied" }
  ]
}

At runtime, conditions are evaluated in order, top to bottom, excluding the last entry. The first one that passes wins. If none pass, the last entry (no if) is taken as the fallback.

Condition syntax

An if string is either a single token or three space-separated tokens:

FormExampleMeaning
<var> has_seal True as soon as the blackboard variable has been set to anything (i.e. is not null) — a simple "flag is set" check.
<var> <op> <int> reputation >= 5 Numeric comparison. Supported operators: <, <=, >, >=, ==, !=. False if the variable is unset.
Unrecognized operators fail closed

Only the six operators above are understood. Anything else in the operator position (a typo, an unsupported symbol) makes the whole condition evaluate to false rather than throwing an error — so a mistyped operator is a silent no-match rather than a crash. A quick manual playtest of that branch is the easiest way to catch that kind of typo.

Blackboard variables

Variable names are discovered automatically — the runtime scans every if string in the loaded file and extracts the first token of each. Values are not part of the dialogue file at all; they live entirely in your game code:

DialogueManager.set_blackboard_variable("reputation", 5)
DialogueManager.set_blackboard_variable("has_seal", true)

Any variable that hasn't been set defaults to null, which makes every condition referencing it evaluate to false — so an unset flag behaves like "off", and an unset numeric variable behaves like "fails every comparison."
