Scripting against DialogueManager

A short tour of every method on the DialogueManager autoload you're likely to call from your own game code. For the signals it emits, see Integration.

Loading and starting

load_dialogue(path: String) -> bool

Reads a dialogue JSON file from disk and parses it. Also scans every if string in the file and pre-registers each blackboard variable name it finds (via read_blackboard_variables() internally), so they exist as null before you set anything. Returns false (and pushes an error) if the file can't be opened or doesn't parse as JSON.

DialogueManager.load_dialogue("res://data/castle_dialogue.json")

start_dialogue(start_node_id: String = "start") -> void

Begins playback from the given node id (defaults to "start"). Emits dialogue_started, then hands control to the per-frame node processor — the first node's line arrives shortly after via line_delivered. Does nothing but push an error if no dialogue has been loaded yet.

Blackboard variables

set_blackboard_variable(variable: String, value) -> void

Sets a value your if conditions can read. Call this any time before (or during) a dialogue — conditions are evaluated live at each choices/conditions node, not snapshotted at load time.

DialogueManager.set_blackboard_variable("reputation", 5)
DialogueManager.set_blackboard_variable("has_seal", true)

read_blackboard_variables(path: String) -> Dictionary

Scans a dialogue file's if strings and returns a {variable_name: null} dictionary of every variable referenced in it. Called automatically by load_dialogue() — you generally don't need to call this yourself, unless you want to inspect what variables a dialogue file expects before loading it (e.g. to build a debug panel).

Advancing the dialogue

advance(choice_index: int = -1) -> void

Moves to the next node. For a plain line, call it with no argument. For a choices or conditions node, pass the index of the choice/branch that was picked (or that passed) — this is what DialogueBox's bundled UI already does for you when a choice button is pressed or a condition passes, so you'll mostly only call this directly if you're writing your own UI instead of using the bundled DialogueBox.

queue_pause_dialogue() -> void

Requests that the dialogue pause at the next safe point (the start of the next node), rather than pausing mid-node. Once that point is reached, dialogue_paused_evt fires and the internal loop stops advancing until you resume it.

continue_dialogue() -> void

Resumes a dialogue previously paused via queue_pause_dialogue(). Does nothing if the dialogue isn't currently paused.

DialogueManager.queue_pause_dialogue()
DialogueManager.dialogue_paused_evt.connect(func():
    await get_tree().create_timer(3.0).timeout
    DialogueManager.continue_dialogue()
)

This pause/resume pair is what lets you interrupt a dialogue for something else in your game (a cutscene, a stealth check, whatever) without tearing down and reloading it — see dialogue/dialogue.gd for the bundled example that does exactly this.

Reading current state

get_current_node() -> Dictionary

Returns the raw node data (as parsed from JSON) for whichever node is currently active — useful for checking .has("choices") or .has("conditions") from your own UI code, the same way DialogueBox.gd does.

get_current_node_id() -> String

Returns the id string of the current node — handy for save/load or debug logging.

eval(condition: String) -> bool

Evaluates a single if-style condition string against the current blackboard values, using the same rules described in Dialogue Format. You generally don't need to call this yourself — DialogueBox.gd already calls it for you to decide which choices to show and which condition branch to take — but it's available if you want to check a condition from your own code without waiting for the dialogue to reach that node.

Not usually called directly

A few methods exist mainly for the bundled DialogueBox to hook into, and you won't normally call them yourself unless you're replacing it with your own UI:

  • register_dialogue_box(box: DialogueBox) — called automatically by DialogueBox._ready() to tell DialogueManager which UI instance to drive. If you write your own dialogue UI from scratch instead of using the bundled one, call this yourself with your own node so the per-node timers (timeout, displaytime) have somewhere to show/hide.
  • dialogue_box_finished_displaying() — a connection point kept for compatibility; it's currently a no-op. The actual auto-advance timing for conditions nodes runs on the node's own displaytime tag inside the internal node processor instead (see Tips for how that timer works).
